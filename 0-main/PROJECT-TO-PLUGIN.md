# 🏭 Промпт: «Преврати проект в агента-плагин»

Четвёртый в семействе рядом с [SKILL-FINDER-PROMPT.md](SKILL-FINDER-PROMPT.md), [PROMPT-ARCHITECT.md](PROMPT-ARCHITECT.md) и [PROMPT-REFACTOR.md](PROMPT-REFACTOR.md).

Запускается **внутри любого проекта**. Анализирует его, задаёт формы, создаёт скиллы, оборачивает в плагин, ставит в Claude Code и Codex, проверяет работоспособность и пишет документацию.

---

## Ресёрч: что нужно знать до того, как строить

Источники: [Claude Code — Skills](https://code.claude.com/docs/en/skills), [Plugins reference](https://code.claude.com/docs/en/plugins-reference), [Agent Skills standard](https://agentskills.io), локальные `plugin-dev`, `skill-creator`, `superpowers:writing-skills`.

### Терминология (её путают чаще всего)

| Сущность | Что это | Контекст | Кто запускает |
|---|---|---|---|
| **Skill** | Папка + `SKILL.md`. Процедурное знание | **общий с тобой** | модель (по `description`) или ты (`/name`) |
| **Agent** | Субагент со своим системным промптом | **изолированный** | модель делегирует |
| **Command** | Плоский `.md`, слэш-команда | общий | только ты |
| **Plugin** | Упаковка: манифест + всё вышеперечисленное | — | устанавливается |

> «Agents are FOR autonomous work, commands are FOR user-initiated actions» — `plugin-dev:agent-development`.

**Практическое следствие:** если стадии пайплайна делят состояние (тема, голос, политики) — это **скиллы**, не агенты. Агент теряет общий контекст. Агенты берут только под изолированную параллельную работу.

### Прогрессивное раскрытие — почему скилл не должен быть толстым

1. **Метаданные** (`name` + `description`) — всегда в контексте (~100 слов)
2. **Тело SKILL.md** — грузится при срабатывании (**< 5k слов**)
3. **`references/`, `scripts/`, `assets/`** — по требованию (практически без лимита; скрипты выполняются вообще без загрузки в контекст)

Правило из `skill-development`: **информация живёт либо в SKILL.md, либо в references — не в обоих**. Дублирование = расхождение версий.

### Корнер-кейсы, на которых всё ломается

| # | Грабли | Правда из документации |
|---|---|---|
| 1 | «Скилл не срабатывает» | Триггер решает **только `description`**. Пиши в третьем лице, ключевой юз-кейс — первым. `description` + `when_to_use` обрезаются на **1536 символах** в листинге |
| 2 | Много скиллов → описания режутся | Листинг имеет бюджет **1% от контекста**. При переполнении описания отбрасываются, начиная с редко используемых. Диагностика — `/doctor`, поднять — `skillListingBudgetFraction` |
| 3 | «Поправил скилл — не применилось» | Текст `SKILL.md` подхватывается **вживую без рестарта**. Но **новая корневая папка скиллов** требует рестарта |
| 4 | «Поправил хук/агента в плагине — не применилось» | `hooks/`, `.mcp.json`, `agents/`, `output-styles/` требуют **`/reload-plugins`** или рестарта |
| 5 | Скилл «перестал влиять» после первого ответа | Содержимое входит в диалог **один раз и остаётся**; файл заново не читается. Пиши **постоянные инструкции**, а не одноразовые шаги |
| 6 | `allowed-tools` «перестал работать» | Грант действует **только на тот ход**, где скилл вызван. Следующее сообщение — грант снят |
| 7 | Плагин не грузится | `plugin.json` **только** в `.claude-plugin/`. Все остальные папки (`skills/`, `agents/`, `commands/`, `hooks/`) — **в корне плагина**. Проверка: `claude plugin validate` |
| 8 | Пути ломаются | Все пути **относительные, начинаются с `./`**. Для абсолютных — `${CLAUDE_PLUGIN_ROOT}` |
| 9 | `/plugin update` говорит «уже последняя» | Ключ кэша — поле `version`. **Не поднял version → апдейта нет**, сколько бы коммитов ни было |
| 10 | Конфликт имён | Приоритет: enterprise → personal → project. Плагинные скиллы живут в неймспейсе `plugin-name:skill-name` и **не конфликтуют** |
| 11 | Проектный плагин не видно | `@skills-dir` проектного скоупа грузится только из `.claude/skills/` **той папки, где запущен** Claude Code, и после диалога доверия. Запускай из корня репозитория |
| 12 | Агент в плагине не стартует | Плагинным агентам **запрещены** `hooks`, `mcpServers`, `permissionMode` — по соображениям безопасности |
| 13 | Сломан YAML во фронтматтере | Скилл загрузится **с пустыми метаданными**: `/name` сработает, а автотриггер — нет. Ловится через `--debug` |

### Два пути упаковки — выбор имеет значение

| Путь | Как | Когда |
|---|---|---|
| **`@skills-dir`** | `claude plugin init <name>` → создаёт `~/.claude/skills/<name>/` с манифестом. Грузится в следующей сессии **без установки и без маркетплейса** | Для себя, для итерации. Самый дешёвый старт |
| **Маркетплейс** | Репозиторий + `.claude-plugin/marketplace.json` → `claude plugin install <name>@<marketplace>` (скоупы `user`/`project`/`local`) | Для раздачи друзьям и команде |

### Переносимость Claude Code ↔ Codex

- **Скилл переносим.** Стандарт [Agent Skills](https://agentskills.io) — это папка + `SKILL.md` с минимумом `name` + `description`. Проверено: Codex использует тот же формат в `~/.codex/skills/<name>/SKILL.md`.
- **Vendor-расширения не переносимы:** `allowed-tools`, `disable-model-invocation`, `context: fork`, `agent`, `model` — специфика Claude Code.
- **Обёртка-плагин** с `agents/`, `hooks/`, `.mcp.json` — механика Claude Code; у Codex своя система плагинов (`~/.codex/plugins/`).
- **Правило:** ядро скилла пиши на общем знаменателе; всё Claude-специфичное выноси так, чтобы его отсутствие не ломало скилл.

---

**Как использовать:** открой агента в папке нужного проекта, скопируй блок ниже, отправь. Он сам вызывает `/skill-creator`.

---

````
/skill-creator

Ты — agent packager. Твоя задача: превратить ЭТОТ проект в агента — набор скиллов,
упакованный в плагин, который можно поставить себе и раздать друзьям.

Работай по фазам. После каждой — короткий отчёт и продолжай, не спрашивая разрешения.
Останавливайся только на Форме (Фаза 1) и на явных развилках.

## Железные правила
1. Ничего не выдумывай: перед упоминанием файла/скилла проверь, что он существует.
2. Не перепиши логику проекта под видом упаковки. Упаковка сохраняет поведение.
3. Один скилл — одна ответственность. Дубли функций схлопывай.
4. Скиллы для стадий, делящих состояние. Агенты — только для изолированной работы.
5. Никаких плейсхолдеров и «остальное аналогично»: каждый файл пиши целиком.

## ФАЗА 0. Разведка проекта
Изучи, что здесь есть, ДО любых вопросов:
- дерево репозитория (глубина 2–3), README, CLAUDE.md / AGENTS.md;
- существующие промпты, воркфлоу, инструкции (`*.prompt.md`, `Prompt*`, `Workflow*`);
- уже созданные скиллы/плагины (`.claude/skills/`, `.claude-plugin/`);
- скрипты и тулинг, которые пайплайн реально запускает;
- артефакты-примеры (готовые выходы) — по ним понятен контракт результата.

Выведи: **Карта проекта** — что делает проект, какие в нём повторяемые процедуры,
какие из них уже формализованы в промптах, что просится в скилл. Отметь, что уже
упаковано и трогать не нужно.

## ФАЗА 1. Форма (задай, дождись ответа)
Покажи форму с ПРЕДЗАПОЛНЕННЫМИ значениями из Фазы 0, чтобы можно было просто
подтвердить. Помечай `(предложено)` там, где заполнил сам.

```
=== АГЕНТ ===
1. Имя агента (kebab-case):        [предложено: ____]
2. Одной фразой — что делает:      [предложено: ____]
3. Кому адресован:                 [предложено: ____]
4. Триггеры (как о нём попросят,
   3–5 фраз живым языком):         [предложено: ____]

=== ГЛАВНЫЙ СКИЛЛ (обязателен) ===
5. Имя:                            [предложено: ____]
6. Что делает от входа до выхода:  [предложено: ____]
7. Что подаётся на вход:           [предложено: ____]
8. Что получается на выходе
   (файлы//структура):             [предложено: ____]
9. Критерий «готово»:              [предложено: ____]

=== ДОПОЛНИТЕЛЬНЫЕ СКИЛЛЫ (необязательно) ===
10. Нужны ли отдельные скиллы под другие функции? (да/нет)
    Если да — для каждого: имя · за что отвечает · вход · выход
    [предложено на основе Фазы 0: ____]

=== РАСПРОСТРАНЕНИЕ ===
11. Только для себя или раздавать? (себе / раздавать)
12. Ставить в Codex тоже? (да/нет)
13. Есть ли настройка под пользователя (voice guide, пути, ключи)? (да/нет)
```

Правила заполнения: если имя не дано — придумай по контексту задачи и явно скажи,
что придумал. Если пункт 10 «нет» — агент состоит из одного скилла, это нормально.

## ФАЗА 2. Проектирование
На основе формы спроектируй состав и покажи ДО создания файлов:
- список скиллов: имя · ответственность · вход · выход · чем отличается от соседей;
- нужны ли агенты (обоснуй изоляцией контекста; по умолчанию — НЕ нужны);
- нужна ли команда-вход (`commands/`) как «одна кнопка»;
- нужны ли скрипты (детерминированные операции лучше выносить в `scripts/`);
- нужен ли скилл настройки под пользователя (`setup-*`), если п.13 = да.

Проверь на дубли: два скилла с одной функцией — ошибка проектирования.

## ФАЗА 3. Создание скиллов
Для каждого скилла — папка и `SKILL.md`:

```
<skill-name>/
├── SKILL.md          # обязателен
├── references/       # детальные материалы, грузятся по необходимости
├── scripts/          # детерминированный код
└── assets/           # шаблоны и файлы для результата
```

Фронтматтер — минимум, переносимый между агентами:
```yaml
---
name: skill-name            # kebab-case, совпадает с именем папки
description: This skill should be used when ... # 3-е лицо, ключевой юз-кейс ПЕРВЫМ
---
```
Claude-специфичное добавляй ТОЛЬКО если оно реально нужно, и помечай комментарием,
что это расширение Claude Code: `allowed-tools`, `disable-model-invocation`,
`context: fork`, `model`, `agent`.

Требования к тексту:
- тело `SKILL.md` — до ~5k слов; детали выноси в `references/`;
- не дублируй одно и то же в SKILL.md и в references;
- пиши ПОСТОЯННЫЕ инструкции: содержимое входит в диалог один раз и заново не читается;
- `description` решает срабатывание. Вставь туда слова, которыми о задаче просят вживую.
  Держись в пределах 1536 символов на `description`;
- если скилл только ручной — `disable-model-invocation: true`.

## ФАЗА 4. Упаковка в плагин
Структура (манифест — только в `.claude-plugin/`, компоненты — в корне):
```
<agent-name>/
├── .claude-plugin/plugin.json
├── skills/<skill>/SKILL.md
├── commands/            # если решили делать вход-кнопку
├── agents/              # только если Фаза 2 обосновала
├── scripts/
└── README.md
```
Манифест (`name` — единственное обязательное поле, kebab-case):
```json
{
  "name": "<agent-name>",
  "version": "0.1.0",
  "description": "<что делает агент>",
  "author": { "name": "<автор>" },
  "keywords": ["..."]
}
```
Все пути в манифесте — относительные, начинаются с `./`. Для абсолютных внутри
скриптов и MCP — `${CLAUDE_PLUGIN_ROOT}`.

Выбор пути упаковки:
- **только себе** → `claude plugin init <agent-name>` создаёт `~/.claude/skills/<agent-name>/`,
  грузится как `<agent-name>@skills-dir` в следующей сессии, без установки;
- **раздавать** → репозиторий + `.claude-plugin/marketplace.json`, установка через
  `claude plugin install <agent-name>@<marketplace>` (скоуп `user` | `project` | `local`).

## ФАЗА 5. Установка
1. Claude Code: выполни установку выбранным способом.
2. Валидация: `claude plugin validate` — проверяет манифест, фронтматтер скиллов
   и агентов, `hooks/hooks.json`. Чини ошибки до зелёного.
3. Codex (если п.12 = да): скиллы переносимы по стандарту Agent Skills. Положи или
   слинкуй каждый скилл в `~/.codex/skills/<skill-name>/SKILL.md`. Claude-специфичные
   поля фронтматтера там игнорируются — убедись, что без них скилл осмысленный.
4. Зафиксируй, что где оказалось (реальные пути, проверенные на диске).

## ФАЗА 6. Проверка работоспособности (доказательства, не утверждения)
1. Файлы: покажи дерево созданного и подтверди, что каждый `SKILL.md` на месте.
2. `claude plugin validate <путь-к-плагину>` прошёл — покажи вывод.
   (Путь обязателен: без аргумента команда не отработает.)
3. `claude plugin details <name>` — покажи инвентарь компонентов и прогноз
   стоимости в токенах. Это ловит две вещи: не подхватился компонент и раздутый
   листинг, который съест бюджет описаний.
4. YAML фронтматтера парсится у КАЖДОГО скилла (сломанный YAML = пустые метаданные
   и молчащий автотриггер).
5. Сухой прогон: возьми один реальный вход из проекта и пройди главный скилл,
   подтвердив, что выход соответствует критерию из п.9 формы.
6. Если проект того стоит — заведи эвалы (`evals/**/case.yaml` либо
   `evals/**/prompt.md` + `graders/*.md`) и прогони `claude plugin eval <name>`.
   Он добавляет baseline-ветку без плагина, то есть показывает, что агент реально
   что-то улучшает, а не просто существует.
7. Если что-то не сошлось — чини файлы, а не отчёт.

## ФАЗА 7. Документация
Создай `README.md` в корне плагина. ПЕРВЫЙ БЛОК — простой и полезный, чтобы человек
за 30 секунд понял, что это и как запустить:

```markdown
# <Agent Name>
<Одна фраза: что делает и для кого.>

## Быстрый старт
1. <установка одной командой>
2. **Перезапусти Claude Code** (или `/reload-plugins`) — иначе новые компоненты не подхватятся.
3. Запусти: `/<главный-скилл>` и передай <что именно>.

## Скиллы
| Скилл | Когда использовать | Вход → Выход |
|---|---|---|
| `/<name>` | ... | ... |

## Флаги и параметры
| Скилл | Флаг | Что делает |
|---|---|---|
```

Дальше — разделы: настройка под себя (если есть `setup-*`), устройство (как скиллы
связаны), troubleshooting, обновление.

Обязательно включи в troubleshooting эти пункты (они реальны и экономят часы):
- правки `SKILL.md` подхватываются вживую; **новая корневая папка скиллов требует рестарта**;
- правки `hooks/`, `agents/`, `.mcp.json` требуют **`/reload-plugins`** или рестарта;
- скилл не срабатывает → дело в `description`: проверь, что там есть слова, которыми
  о задаче просят вживую; проверь список через «какие скиллы доступны?»;
- много скиллов → описания режутся бюджетом листинга, диагностика `/doctor`;
- обновление у пользователей не приезжает → **подними `version` в `plugin.json`**;
  после `claude plugin update` нужен **рестарт**, чтобы новая версия применилась;
- проектный `@skills-dir` плагин виден только при запуске из корня репозитория;
- полезные команды для диагностики: `claude plugin list` (что стоит),
  `claude plugin details <name>` (состав + стоимость в токенах),
  `claude plugin validate <path>` (манифест и фронтматтер).

## Финальный отчёт
Выведи компактно:
- имя агента и что он делает;
- список созданных скиллов с назначением;
- куда установлено (реальные пути, проверенные на диске);
- результаты Фазы 6 (что именно проверено и чем доказано);
- как проверить руками: 3 команды, которые пользователь выполнит прямо сейчас;
- как раздать друзьям;
- что осталось сделать и почему.
````
