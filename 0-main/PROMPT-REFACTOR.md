# 🧩 Промпт: «Разбери монолитный промпт на модули + оркестратор»

Третий в семействе рядом с [SKILL-FINDER-PROMPT.md](SKILL-FINDER-PROMPT.md) (подбор скиллов) и
[PROMPT-ARCHITECT.md](PROMPT-ARCHITECT.md) (задача → промпт). Этот делает **рефакторинг промптов**:
берёт один разросшийся промпт и превращает его в набор модулей с мастер-оркестратором.

## На чём основано (ресёрч)

| Источник | Что взято |
|---|---|
| [Anthropic — Building Effective Agents](https://www.anthropic.com/research/building-effective-agents) | 5 паттернов (chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer), критерии «когда НЕ применять», gates между шагами, принцип простоты |
| [Anthropic — Prompting best practices](https://platform.claude.com/docs/en/docs/build-with-claude/prompt-engineering/chain-prompts) | XML-теги для однозначной структуры, self-correction цепочка (draft → review → refine), правило анти-оверинжиниринга |
| [Modular Prompt Engineering 2026](https://chatpromptgenius.com/modular-prompt-engineering-best-practices-for-2026/) · [Prompt Composition](https://www.autolearningagents.com/ai-agent-architecture/prompt-composition.php) | Композиция из базового слоя + модификаторов, сборка на рантайме, условное включение модулей, версионирование фрагментов |

**Три вывода, которые встроены в промпт как правила:**
1. **Не дроби ради дробления.** Anthropic прямо пишет: явное разбиение на цепочку нужно, «когда тебе
   надо инспектировать промежуточные выходы или зафиксировать структуру пайплайна». Если этого нет —
   монолит может быть правильным ответом. Цепочка **меняет латентность на точность**, это осознанная сделка.
2. **Если подзадачи взаимозависимы — цепочка противопоказана.** Разбивать можно только то, что режется чисто.
3. **Одно правило живёт ровно в одном модуле.** Дубль правила в двух модулях убивает весь смысл рефакторинга:
   правку придётся вносить в двух местах, и они разъедутся.

**Как использовать:**
1. Запусти агента в папке проекта, где лежит твой монолитный промпт.
2. Скопируй блок ниже — он сам вызывает `/prompt-engineering`.
3. Укажи путь к промпту в `МОНОЛИТ:` и отправь.

---

````
/prompt-engineering

Ты — prompt systems architect. Твоя задача — провести РЕФАКТОРИНГ одного разросшегося
промпта: разложить его на модули с одной ответственностью и собрать мастер-оркестратор,
который прогоняет их по цепочке. Это инженерная работа, а не переписывание текста.

## Железные принципы (не нарушай)
1. **Ни одно правило не теряется.** Каждое требование исходного промпта должно оказаться
   ровно в ОДНОМ модуле. Потеря правила = провал рефакторинга.
2. **Одно правило — один дом.** Если правило продублировано в двух модулях, ты не улучшил
   поддерживаемость, а удвоил её стоимость. Дубли схлопывай в общий модуль.
3. **Поведение не меняется.** Рефакторинг сохраняет результат. Если по дороге ты увидел, как
   улучшить логику, — НЕ делай этого, вынеси в отдельный список «Предложения на потом».
4. **Простота важнее красоты.** Не создавай модуль ради симметрии. Минимальное число модулей,
   при котором каждый имеет свою ответственность и свой темп изменений.
5. **Дроби только то, что режется чисто.** Взаимозависимые подзадачи в цепочку не выносятся.

## ФАЗА 0. Инвентаризация монолита (до всякого проектирования)
Прочитай исходный промпт и построй **реестр правил**. Пронумеруй КАЖДОЕ требование:
| # | Правило (цитата/суть) | Категория | Когда применяется |
Категории: `Роль/тон` · `Доменные знания` · `Процесс/шаги` · `Контракт вывода` ·
`QA-гейты` · `Крайние случаи` · `Запреты`.
В конце укажи: всего правил N, распределение по категориям.
Это твой baseline для аудита покрытия в Фазе 6.

## ФАЗА 1. Решение: дробить или нет (честно)
Разбиение оправдано, если выполняется хотя бы одно:
- нужно **инспектировать промежуточный результат** (логировать, валидировать, ветвиться);
- нужно **зафиксировать структуру пайплайна**, а не полагаться на внутреннее рассуждение модели;
- в промпте живут **конфликтующие наборы правил** (разные подзадачи тянут в разные стороны);
- у частей **разный темп изменений** (одна правится еженедельно, другая — раз в год);
- часть логики **переиспользуется** в других проектах;
- монолит настолько велик, что модель **теряет правила из середины**.

Разбиение ПРОТИВОПОКАЗАНО, если: подзадачи взаимозависимы и не режутся чисто; выигрыш только
косметический; цена латентности выше выигрыша в точности.

Вынеси вердикт явно: **ДРОБИМ / НЕ ДРОБИМ / ДРОБИМ ЧАСТИЧНО** с обоснованием.
Если «не дробим» — так и скажи и предложи вместо этого навести порядок внутри монолита
(секции, XML-теги, вынос справочников в отдельные файлы). Не выдумывай работу.

## ФАЗА 2. Выбор топологии
Выбери схему и обоснуй, почему не остальные:
- **Prompt chaining** — фиксированная последовательность шагов, выход одного = вход следующего.
  Базовый выбор для «сделать X в 5 этапов». Меняет латентность на точность.
- **Routing** — сначала классификация входа, потом специализированный обработчик.
  Когда типы входа принципиально разные (не пихай их в один промпт).
- **Parallelization** — независимые части считаются одновременно (sectioning) либо одна задача
  прогоняется несколько раз для надёжности (voting). Только если части реально независимы.
- **Orchestrator-workers** — центральный модуль сам решает, какие подзадачи нужны.
  Когда состав подзадач заранее НЕ известен.
- **Evaluator-optimizer** — генератор + критик в цикле. Когда есть внятные критерии качества
  и итерации измеримо улучшают результат.
Паттерны композируются (routing → chain; в chain-шаге может сидеть evaluator-optimizer).

## ФАЗА 3. Границы модулей
Режь по принципу: **одна ответственность + свой набор правил + свой темп изменений**.
Для каждого модуля выпиши карточку-контракт:
```
### M{n}. {имя}
Назначение:      одна фраза — что он делает и чего НЕ делает
Владеет правилами: #3, #7, #12-15 (номера из реестра Фазы 0)
Вход:            имя + схема/формат + пример
Выход:           имя + схема/формат + пример
Гейт (проверка): что должно быть истинно на выходе, иначе шаг считается провалившимся
Отказ:           что делать при провале (retry / вернуть на шаг назад / остановиться)
Темп изменений:  часто / редко
```

## ФАЗА 4. Контракт передачи данных
- Каждый handoff — **структурированный**: JSON по схеме либо XML-теги
  (`<input>`, `<context>`, `<output>`). Свободная проза между модулями запрещена: она
  порождает дрейф интерпретации.
- Явно перечисли, что модуль обязан пробросить дальше **без изменений** (сквозной контекст:
  бренд, локаль, ID) и что он вправе трансформировать.
- Между шагами ставь **программные гейты**: дешёвая детерминированная проверка (схема
  валидна, счётчики сходятся, обязательные поля на месте) до передачи дальше.

## ФАЗА 5. Мастер-оркестратор
Напиши сам оркестратор-промпт. Обязательные части:
1. Реестр модулей: порядок, файл каждого, назначение.
2. Правила исполнения: строго по очереди; выход шага N — вход шага N+1; ничего не пропускать.
3. Гейты и что делать при провале (сколько ретраев, когда откатываться, когда звать человека).
4. Сквозное состояние: что живёт от начала до конца.
5. Логирование: на каждом шаге — что вошло, что вышло, гейт пройден/провален.
6. Финальная сборка: как из выходов модулей собирается итоговый артефакт.
7. Отчёт: пройденные шаги, провалы, итоговые файлы.

## ФАЗА 6. Аудит покрытия (гейт качества, обязателен)
Таблица «правило → модуль» по ВСЕМУ реестру Фазы 0:
| # правила | Категория | Модуль | Статус |
Статус: `Перенесено` · `Объединено с #X` · `Продублировано → схлопнуто в M{n}` ·
`Устарело (обосновать)` · `НЕ перенесено`.
**Любое `НЕ перенесено` блокирует сдачу.** Счётчики обязаны сходиться: правил на входе = правил
на выходе (минус явно обоснованные устаревшие). Приведи арифметику.

## ФАЗА 7. Проверка, что не сломалось
- Собери **golden set**: 3–5 реальных входов с известным хорошим результатом.
- Прогони их через СТАРЫЙ монолит и НОВУЮ цепочку, сравни результаты по пунктам.
- Регресс = провал рефакторинга, а не «новая версия». Чини цепочку.
- Заведи мини-эвалы на модуль: для каждого — 2–3 входа и ожидаемая форма выхода.

## ФАЗА 8. Версионирование и эксплуатация
Предложи раскладку и правила сопровождения:
```
prompts/
  orchestrator.md          # мастер
  modules/
    01-{имя}.md
    02-{имя}.md
  contracts/schemas.md     # схемы входов/выходов
  evals/golden-set.md
  RULES-MAP.md             # правило → модуль (из Фазы 6)
  CHANGELOG.md
```
Плюс таблица **«хочу изменить X → правь файл Y»** — ради неё всё и затевалось.

## Формат ответа
1. **Вердикт Фазы 1** (дробим/нет + почему) и **выбранная топология** с обоснованием.
2. **Карта модулей** — схема потока (текстом: `вход → M1 → гейт → M2 → … → выход`).
3. **Файлы** — полное содержимое каждого модуля и оркестратора, готовое к сохранению.
   Никаких «...остальное аналогично»: пиши каждый модуль целиком.
4. **Аудит покрытия** (Фаза 6) со сходящимися счётчиками.
5. **Golden set и как прогнать** (Фаза 7).
6. **Таблица сопровождения** «изменить X → файл Y».
7. **Предложения на потом** — улучшения логики, которые ты СОЗНАТЕЛЬНО не внёс.

## Запреты
- Не улучшать логику под видом рефакторинга.
- Не терять правила и не дублировать их между модулями.
- Не плодить модули ради симметрии; не проектировать под гипотетическое будущее.
- Не отдавать результат без сошедшегося аудита покрытия.
````
